# Implementation Plan: Chat UI (Upload + Streaming Q&A)

**Branch**: `002-chat-ui` | **Date**: 2026-05-17 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/002-chat-ui/spec.md`

## Summary

A React + TypeScript SPA styled with Tailwind CSS, built inside the existing
root multi-stage Dockerfile and served as static assets by the FastAPI backend
on the same origin. The SPA exercises the existing `/upload`, `/ask` (SSE),
`/history/{session_id}`, and `/session/end` routes shipped in feature 001.
Streaming is consumed with `fetch` POST + `ReadableStream` (not `EventSource`),
cancellation via `AbortController`. Session handle persisted in `sessionStorage`
per tab and rehydrated on reload through `GET /history/{session_id}`. The SPA
attaches no `Authorization` header; the backend trusts same-origin browser
requests, with a dedicated ADR capturing the threat model.

## Technical Context

**Language/Version**: TypeScript 6+ (SPA); Python 3.11 (existing backend, no
language change in this feature).

**Primary Dependencies** (latest stable at 2026-05-17; exact versions
pinned in `frontend/package.json` and recorded here as the floor):

- SPA runtime: React 19.x (`react`, `react-dom` ≥ 19.0), Tailwind CSS
  4.x (CSS-first config via `@import "tailwindcss";` + `@theme {}`;
  no `tailwind.config.{js,ts}`), `clsx` 2.x (class merging only — no
  CSS-in-JS), `eventsource-parser` 3.x (SSE frame parsing on top of
  `fetch` + `ReadableStream`). No state library (React 19's `useReducer`
  + `useContext` are sufficient for the session/transcript machine;
  Redux/Zustand rejected as overkill).
- SPA build/dev: Vite 8.x, `@vitejs/plugin-react` (latest matching Vite
  8), `@tailwindcss/vite` 4.x plugin (replaces the v3 PostCSS pipeline),
  TypeScript 6.x.
- SPA test: Vitest 4.x (4.1.6+), `@testing-library/react` (React
  19-compatible release), `@testing-library/jest-dom` (latest), MSW
  2.x, `openapi-typescript` 7.x (generates typed client from
  `openapi.yaml`). **No Playwright / no headless browser**. The
  consumer-side streaming-incrementality assertion is implemented as a
  Vitest test that drives the production SPA client (`api/ask.ts`)
  against an MSW 2 streaming handler emitting SSE frames with deliberate
  awaits between them, asserting ≥ 2 `token` dispatches arrive before
  `done`. JSDOM is the test environment.
- Backend: no new runtime deps. Static-file serving via FastAPI's
  existing `StaticFiles` mount.
- Build: Vite 8 production build invoked from the existing
  `frontend-builder` stage of the root `Dockerfile`; output
  (`/frontend/dist`) copied into the runtime stage and mounted at `/`
  by FastAPI.

> (`^19.0.0`, `^8.0.0`, `^4.0.0`, …) and `package-lock.json` MUST be
> committed. Upgrading any of the five marquee packages (React, Vite,
> Tailwind, TypeScript, Vitest) across a major version requires a plan
> amendment, not a silent bump.
- Backend: no new runtime deps. Static-file serving via FastAPI's existing
  `StaticFiles` mount.
- Build: Vite production build invoked from the existing `frontend-builder`
  stage of the root `Dockerfile`; output (`/frontend/dist`) copied into the
  runtime stage and mounted at `/` by FastAPI.

**Storage**: Browser `sessionStorage` (per-tab, ephemeral) for the active
`session_id` only. No `localStorage`. No cookies (backend has none). No new
backend storage.

**Testing**:

- SPA unit/component: Vitest 4 + React Testing Library (React 19
  compatible), JSDOM environment, run inside the `frontend-dev`
  container (`docker compose run --rm frontend-dev npm test`).
- SPA contract: typed client generated by `openapi-typescript` 7
  against `specs/001-doc-assistant-rag/contracts/openapi.yaml` — tests
  assert the client compiles against the schema and that an MSW 2 fake
  of each route is handled.
- SPA streaming integration (replaces the dropped Playwright layer):
  Vitest test that drives `api/ask.ts` against an MSW 2 streaming
  handler, asserts ≥ 2 incremental `token` events dispatched to the
  reducer before `done`. This satisfies the constitution's
  streaming-incrementality gate on the consumer side without a
  headless browser.
- No Playwright. The backend already owns the end-to-end HTTP
  streaming test (feature 001); the consumer-side assertion above
  pairs with it.

**Target Platform**: Modern evergreen browsers (Chromium ≥ last 2, Firefox
≥ last 2, Safari ≥ last 2) on viewports ≥ 600 px wide. Served by the same
Linux container that runs FastAPI.

**Project Type**: Web application (existing backend + new frontend) — repo
adopts the "frontend/ + src/" layout already scaffolded.

**Performance Goals**:

- First visible answer token within 2 s of submit (p95) — inherits
  constitution Principle IV and spec FR-009 / SC-002.
- Production bundle (gzipped) ≤ 200 KB main + 100 KB Tailwind CSS. Justified
  because React + Tailwind already pay a fixed cost; we cap drift.
- 60 fps token-render: no layout thrash during streaming a 500-word answer
  (SC-004). Achieved by appending to a stable text node, not re-rendering
  the whole transcript per token.

**Constraints**:

- No `EventSource` (cannot send `Authorization` if we later add header
  injection, no POST body) — `fetch` + `ReadableStream` only (FR-026).
- No CSS-in-JS, no Bootstrap, no global CSS beyond Tailwind base (FR-020).
- No secrets in the bundle (FR-024). `import.meta.env.VITE_*` MUST NOT be
  used to embed any token.
- `docker compose up` MUST remain the only documented start path (FR-022).

**Scale/Scope**: Single-tenant demo. One concurrent session per tab.
Transcript bounded by the existing backend in-memory cap (`MAX_HISTORY_TURNS`
from feature 001). UI MUST render up to that cap without virtualization
(measured headroom; if exceeded later, swap to `react-virtuoso`).

## Constitution Check

*Re-check after Phase 1 design.* All five principles addressed below.

| # | Principle | How this feature satisfies it |
|---|-----------|-------------------------------|
| I | Code Quality & Clean Architecture | SPA is layered: `api/` (typed HTTP + SSE client) → `state/` (session reducer, no business logic in components) → `components/` (presentational) → `pages/`. No orchestration framework (no Redux, no LangChain-equivalent). All cross-layer calls go through typed interfaces (e.g., `ChatClient`, `StreamReader`). Strict TS (5.8) + `tsc --noEmit`, ESLint 9 (flat config), and Prettier in CI; failures block merge. No new backend layers added — feature reuses the five existing interfaces (`DocumentParser`, `EmbeddingProvider`, `VectorStore`, `LLMClient`, `ConversationStore`). |
| II | Test-First Discipline | Each SPA module ships with its failing test first (Vitest). Streaming-cancel test asserts `AbortController.abort()` halts the reader and the backend `done` event is not appended. E2E test asserts ≥ 2 incremental token frames (matches constitution streaming gate). Contract test compiles the typed client against `openapi.yaml` and exercises MSW mocks for every route the SPA uses. |
| III | UX Consistency | Spec enumerates idle / uploading / streaming / error states explicitly (FR-002, FR-005, FR-012, FR-013, FR-014). Behavior is provider-agnostic (the SPA never reads `LLM_PROVIDER` / `EMBEDDING_PROVIDER`). Errors render inline in the chat surface, never as alerts or stack traces. |
| IV | Performance & Streaming | Streaming consumed incrementally via `fetch` + `ReadableStream`; first token ≤ 2 s p95 (FR-009). Transcript append uses a ref-mutating text node during streaming (commit once on `done`) to avoid per-token React reconciliation across the whole list. No blocking I/O on the main thread; large file reads stay in `<input type=file>` / `FormData` (browser-native, off-main-thread). |
| V | Configuration-Driven Extensibility | Only one FE-side config knob: `VITE_API_BASE_URL` (defaults to same-origin `""` in production, `http://app:8000` in the dev container). Documented in `.env.example`. No FE branching on provider — provider swap remains an env-only backend change, fully transparent to the SPA. |

**Gate verdict**: PASS. No violations to record in Complexity Tracking.

### Specific risks the spec flags that this plan resolves

- **R1 — Auth model contradiction (spec FR-025 vs. existing OpenAPI bearer
  requirement).** Resolved in `research.md` decision *Auth posture for
  browser-originated requests*. Backend MUST add an "auth bypass for
  same-origin browser requests when the request carries the static-asset
  origin" rule; non-browser callers continue to require bearer. ADR captures
  the tradeoff. No change to provider keys or `APP_SHARED_TOKEN` semantics.
- **R2 — SSE parsing edge cases (multi-line `data:`, comments, retry).**
  Use the well-tested `eventsource-parser` package; do not hand-roll the
  state machine. Justified runtime dependency under constitution's
  "convenience wrapper" rule because it is a *protocol* parser, not a
  convenience wrapper around stdlib.
